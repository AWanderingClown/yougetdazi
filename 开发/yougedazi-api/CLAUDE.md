# 有个搭子 - 后端API

## 项目简介
线下陪玩搭子平台的后端API服务。技术栈：Fastify + Prisma + PostgreSQL + Redis + BullMQ + Socket.IO。

### 架构原则
- **后端计算，前端展示**：所有业务逻辑集中在后端，前端仅负责渲染和交互
- **服务拆分**：按职责拆分独立服务，降低耦合
- **软编码优先**：可通过管理后台动态配置，减少代码发布

### 后端服务器 (yougedazi-api)

**核心职责**：
- 所有业务逻辑计算（订单费用、退款金额、距离计算等）
- 数据验证和权限控制
- 数据库操作（Prisma ORM）
- 对外提供 RESTful API
- Webhook回调处理（微信支付等）

**不处理**：
- 页面渲染
- 用户交互逻辑
- 样式计算

**技术栈**：
- Node.js 20 + Fastify 5
- Prisma ORM 7 (with driverAdapters preview)
- PostgreSQL 15
- Redis 7

## 核心约束（必须遵守）
直接修改现有文件，不要创建新文件，不要生成markdown说明文档，不要写对比报告。如果必须解释，用代码注释而非外部文件。保持项目现有目录结构

### 修改前必须做的
1. 只能修改指定的文件，其他文件一律不许动
2. 修改前必须列出要改的文件清单，等用户确认后再编码
3. 不要删除现有代码（只能新增或修改）

## 编码注意事项

### 编辑文件前
1. **必须完整读取目标文件**，不能只读部分内容就进行编辑
2. 选择oldString时：
   - 必须足够长（至少5行以上）
   - 必须唯一（避免选择重复模式的代码）
   - 必须与文件中的**实际内容**完全一致（注意缩进、空格、换行符）
3. 编辑后立即验证：运行 `npm run build` 确认编译通过

### 常见错误原因
| 错误 | 原因 | 预防 |
|------|------|------|
| oldString not found | 读取的内容与文件实际内容有偏差 | 完整读取后再选择代码片段 |
| Found multiple matches | 选择的代码在文件中多处出现 | 选择更长的唯一片段 |
| invalid type | newString参数缺失或格式错误 | 检查JSON格式是否正确 |
| 路径拼写错误 | 复制路径时出错 | 仔细核对路径 |

### 最佳实践
- 修改前先用 `read()` 完整查看文件内容
- 选择oldString时，包含足够的上下文（函数签名、关键变量等）
- 避免选择包含常见模式（如 `if (!parseResult.success)`）的短代码块
- 复杂修改建议分步进行，每次只改一个逻辑点
